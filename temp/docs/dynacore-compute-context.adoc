= RFC: DynaCore ComputeContext and Cell-Centric Execution
:toc: left
:toclevels: 3
:icons: font

== Status

*Stage:* Draft

*Decision posture:* Adopt only after real profiling confirms allocations are a primary bottleneck.

*Policy:* Hard compile-time migration break. No compatibility layer.

== Context

Per-tick heap allocations in xref:glossary.adoc#domain-dynacore[DynaCores] (multiple `Dictionary<CellId, double>` instances)
create GC pressure at larger grid sizes (for example 10k to 100k cells per xref:glossary.adoc#engine-domain[Domain]).

Current boundary-centric iteration also blocks straightforward parallel execution because a
single boundary updates two cells, producing shared-write contention.

The objective is to move to a xref:glossary.adoc#topology-cell[Cell]-centric kernel model that is allocation-conscious,
parallel-safe, and easier to scale.

== Goals

- Reduce per-tick allocations across xref:glossary.adoc#domain-dynacore[DynaCores].
- Enable safe intra-tick parallelism.
- Preserve deterministic behavior.
- Keep xref:glossary.adoc#domain-component[Components] and xref:glossary.adoc#domain-dynacore[DynaCores] stateless in architecture.

== Non-Goals

- Introducing runtime compatibility shims for legacy `Compute(...)` contracts.
- Migrating before benchmark data exists.
- Changing xref:glossary.adoc#domain-state[State] semantics of current physical fields.
- Changing xref:glossary.adoc#domain-forcing[Forcing] or xref:glossary.adoc#engine-cadence[Cadence] semantics.

== Core Proposal

=== 1. ComputeContext replaces raw grid argument

Replace the xref:glossary.adoc#topology-grid[Grid] parameter on `IDynaCore.Compute` with a context object containing
compute-time resources.

[source,csharp]
----
public sealed class ComputeContext(IGrid grid, CellIndex cellIndex);
----

This centralizes context growth and avoids repeated signature churn.

=== 2. CellIndex provides stable ordinals and scratch fields

`CellIndex` maps xref:glossary.adoc#topology-cell[Cell] identifiers (`CellId`) to stable ordinals for a given `(xref:glossary.adoc#engine-domain[Domain], CellSet)`.

[source,csharp]
----
public sealed class CellIndex
{
        public int Ordinal(CellId cell);
        public CellId CellAt(int ordinal);
        public double[] RentField();
        public void ReturnField(double[] buffer);
}
----

The backing fields become array-based and ordinal-indexed.

=== 3. CellIndex provisioning via registry

xref:glossary.adoc#domain-component[Components] remain stateless and do not own compute resources.

An external registry (for example `ICellIndexRegistry`) manages:

- index creation
- index rebuild on cell-set change
- pooled field lifetime
- stable ordinal mapping guarantees

[source,csharp]
----
var cellIndex = cellIndexRegistry.Get(context.Domain, context.CellSet);
----

=== 4. Cell-centric execution model

Move from boundary-centric updates:

[source,text]
----
for boundary:
        update(left)
        update(right)
----

to cell-centric updates:

[source,text]
----
for each cell:
        for each neighbor:
                accumulate contribution
        update(cell)
----

xref:glossary.adoc#domain-component[Component]-level parallelism becomes viable:

[source,csharp]
----
Parallel.For(0, cellCount, i =>
{
        dynaCore.ComputeCell(...);
});
----

=== 5. DynaCore contract becomes cell-kernel oriented

`Compute(...)` is replaced by `ComputeCell(...)`.

[source,text]
----
ComputeCell(context, cellIndex, neighbors, fields...) -> updates one cell
----

== Explicit Invariants

=== Invariant A: Single-writer per cell

Each cell ordinal is written exactly once per tick.

Consequences:

- no write conflicts
- no locks required for field writes
- predictable parallel scaling

=== Invariant B: Stateless DynaCore behavior

xref:glossary.adoc#domain-dynacore[DynaCore] instances own no mutable long-lived resources.

All mutable compute resources are supplied through `ComputeContext` and returned to pools by
the orchestration layer.

=== Invariant C: Ownership boundaries

xref:glossary.adoc#domain-component[Component] orchestration owns traversal and scheduling.
xref:glossary.adoc#domain-dynacore[DynaCore] owns local physics/kernel math only.

=== Invariant D: Deterministic execution requirements

- Fixed neighbor ordering per cell.
- Local accumulation per cell before writeback.
- No unordered shared reductions inside parallel regions.

=== Invariant E: Interaction symmetry rules

Cell-centric iteration can evaluate pair interactions twice (`A -> B` and `B -> A`).

For symmetric pair terms, apply `weight = 0.5` to preserve equivalence with boundary-centric totals.

== Contract Rules

=== Allowed inside `ComputeCell`

- read current cell
- read neighbor cells
- read xref:glossary.adoc#domain-parameter[Parameters] and previous xref:glossary.adoc#domain-state[State]
- write current cell only

=== Forbidden inside `ComputeCell`

- writing to neighbor cells
- direct topology traversal through `IGrid`
- per-cell heap allocation

== Migration Path (Hard Break)

This migration is intentionally compile-time breaking.

No compatibility layer (`Compute` fallback, adapters, runtime switching) is introduced.
The current codebase has one real xref:glossary.adoc#domain-dynacore[DynaCore] benchmark reference, so fail-fast compiler guidance is
preferred over latent runtime divergence.

=== Migration steps

1. Replace `IDynaCore.Compute(...)` with `ComputeCell(...)` in Brunnr contracts.
2. Update BaseComponent orchestration to cell-centric execution.
3. Introduce `ComputeContext` and `ICellIndexRegistry` provisioning.
4. Port benchmark xref:glossary.adoc#domain-dynacore[DynaCore] first.
5. Treat compile errors as migration checklist for remaining test doubles and components.

== Phase Plan

=== Phase 0: Measure first

Gate work on real profiling:

- collect `tectonics.performance.compute-ms`
- capture allocation profile under representative xref:glossary.adoc#topology-grid[Grid] and xref:glossary.adoc#engine-domain[Domain] counts
- confirm allocation pressure is a top contributor

=== Phase 1: Infrastructure introduction

- add `ComputeContext`
- add `CellIndex` + registry lifecycle
- wire retrieval by `(xref:glossary.adoc#engine-domain[Domain], CellSet)`

=== Phase 2: Contract break and benchmark migration

- hard-break `IDynaCore`
- move benchmark xref:glossary.adoc#domain-dynacore[DynaCore] to `ComputeCell`
- enforce single-writer and deterministic ordering invariants

=== Phase 3: Generalization

- migrate additional xref:glossary.adoc#domain-dynacore[DynaCores] as they are introduced
- tune pooling, neighbor caching, and execution policy

== Impact

- `IDynaCore` API changes (`Compute -> ComputeCell`)
- no backward-compatibility adapter for old signatures
- `BaseComponent` gains responsibility for:
    - cell-level parallel execution
    - neighbor resolution and neighbor cache ownership
    - compute-resource provisioning via registry
- `CellIndex` lifecycle becomes first-class infrastructure

== Risks and Mitigations

- Risk: orchestration bloat in `BaseComponent`.
        Mitigation: keep physics logic exclusively in xref:glossary.adoc#domain-dynacore[DynaCore] kernel helpers.

- Risk: nondeterminism from parallel ordering.
    Mitigation: enforce deterministic neighbor ordering and single-writer pattern.

- Risk: hidden conversion overhead from arrays to external dictionary views.
    Mitigation: define projection policy explicitly (cached/snapshot/lazy) before rollout.

== Open Questions

- Should neighbor adjacency be cached per `(xref:glossary.adoc#engine-domain[Domain], CellSet)` in the same registry layer?
- Should field projection to external state dictionaries be eager or lazy?
- Should execution use `Parallel.For` directly or a scheduler abstraction for future tuning?
